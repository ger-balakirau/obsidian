# Автоматический ввод ПК в домен с флешки

Эта заметка про практический сценарий: после установки Windows быстро ввести компьютер в нужный домен без ручного прохода через настройки системы.

Есть два подхода:

1. Скрипт `join-domain.ps1` - проще, но спросит доменный пароль.
2. Offline Domain Join через `djoin` - лучше для массовой подготовки, пароль на клиенте вводить не нужно.

## Когда это нужно

- Быстрая подготовка новых рабочих мест.
- Лабораторный стенд с AD.
- Замена ПК пользователю.
- Подготовка партии компьютеров под один домен и OU.
- Ситуация, когда не хочется каждый раз руками открывать настройки Windows.

## Что должно быть готово

- ПК подключён к сети, где виден контроллер домена.
- DNS на клиенте указывает на DNS контроллера домена.
- Время на клиенте не сильно отличается от DC.
- Есть OU, куда нужно положить компьютер.
- Есть учётка с правом вводить ПК в домен или заранее созданный `djoin`-файл.

Как настроить такую учётку: [[17 - Настройка пользователя joinuser для ввода ПК в домен]].

Проверочные команды:

```cmd
ipconfig /all
ping lab.loc
nslookup lab.loc
nltest /dsgetdc:lab.loc
```

## Запуск после установки Windows

На экране первой настройки Windows можно открыть консоль:

```text
Shift + F10
```

Запустить PowerShell:

```cmd
powershell
```

Разрешить запуск скрипта на текущий сеанс:

```powershell
Set-ExecutionPolicy Bypass -Scope Process -Force
```

Найти букву флешки:

```cmd
diskpart
list volume
exit
```

Запустить скрипт:

```powershell
D:\join-domain.ps1
```

`D:` - пример. На реальном ПК флешка может быть `E:`, `F:` или другой буквой.

## Вариант 1 - PowerShell join-domain.ps1

Файл на флешке:

```text
D:\join-domain.ps1
```

Пример скрипта с комментариями:

```powershell
# Имя компьютера, которое будет видно в домене и в AD.
# Лучше заранее договориться о шаблоне имён: SKLAD1, OFFICE-001, KASSA-01.
$ComputerName = "SKLAD1"

# DNS-имя домена Active Directory.
$Domain = "lab.loc"

# OU, куда нужно поместить компьютер после ввода в домен.
# Путь пишется в LDAP-формате: OU=...,DC=...
$OU = "OU=Workstations,OU=Kaluga,DC=lab,DC=loc"

# IP-адрес DNS-сервера домена. Обычно это контроллер домена.
# Без правильного DNS клиент может не найти домен.
$DnsServer = "192.168.1.10"

# Учётная запись, которой разрешено вводить компьютеры в домен.
# Пароль будет запрошен интерактивно, хранить его в скрипте не нужно.
# Эту учётку нужно заранее делегировать на нужную OU.
$JoinAccount = "LAB\joinuser"

# Находим активные сетевые адаптеры.
# Если подключено несколько сетей, скрипт применит DNS ко всем активным адаптерам.
$ActiveAdapters = Get-NetAdapter | Where-Object { $_.Status -eq "Up" }

# Ставим DNS контроллера домена на активные сетевые адаптеры.
# Это критично для поиска DC и корректной работы AD.
foreach ($Adapter in $ActiveAdapters) {
    Set-DnsClientServerAddress `
        -InterfaceIndex $Adapter.InterfaceIndex `
        -ServerAddresses $DnsServer
}

# Проверяем, что домен находится через DNS.
# Если здесь ошибка, join почти наверняка не сработает.
nslookup $Domain

# Переименовываем компьютер.
# После join и перезагрузки в AD будет нормальное имя, а не DESKTOP-XXXX.
Rename-Computer -NewName $ComputerName -Force

# Вводим компьютер в домен и кладём его в нужную OU.
# Команда спросит пароль от $JoinAccount.
# После успешного join компьютер автоматически перезагрузится.
Add-Computer `
    -DomainName $Domain `
    -OUPath $OU `
    -Credential $JoinAccount `
    -Restart
```

Запуск:

```powershell
Set-ExecutionPolicy Bypass -Scope Process -Force
D:\join-domain.ps1
```

После запуска Windows спросит пароль от `LAB\joinuser`, выполнит join и перезагрузится.

## Вариант 2 - Offline Domain Join через djoin

`djoin` позволяет заранее подготовить файл присоединения к домену. На новом ПК не нужно вводить доменный логин и пароль.

Это хороший вариант для флешки и массовой подготовки рабочих мест.

## Шаг 1 - создать ODJ-файл на контроллере домена

На DC или на админской машине с RSAT:

```cmd
djoin /provision /domain lab.loc /machine SKLAD1 /machineou "OU=Workstations,OU=Kaluga,DC=lab,DC=loc" /savefile C:\ODJ\SKLAD1.txt
```

Что делает команда:

```text
/provision   - заранее создаёт объект компьютера в домене
/domain      - домен, куда будет вступать ПК
/machine     - имя будущего компьютера
/machineou   - OU, куда положить компьютер
/savefile    - файл, который потом переносится на новый ПК
```

Файл:

```text
C:\ODJ\SKLAD1.txt
```

нужно положить на флешку.

Важно: ODJ-файл предназначен для конкретного имени компьютера. Для другого ПК лучше создать отдельный файл.

## Шаг 2 - применить ODJ-файл на новом ПК

На новом ПК после установки Windows:

```cmd
Shift + F10
```

Найти флешку:

```cmd
diskpart
list volume
exit
```

Применить файл:

```cmd
djoin /requestodj /loadfile D:\SKLAD1.txt /windowspath C:\Windows /localos
shutdown /r /t 0
```

Что делает команда:

```text
/requestodj       - применяет offline domain join
/loadfile         - путь к заранее созданному ODJ-файлу
/windowspath      - путь к установленной Windows
/localos          - применить к текущей локальной ОС
shutdown /r /t 0  - сразу перезагрузить ПК
```

После перезагрузки компьютер уже будет присоединён к домену.

## Удобный join.cmd для флешки

На флешке можно держать простой `join.cmd`.

Файл:

```text
D:\join.cmd
```

Содержимое:

```cmd
@echo off

rem Имя ODJ-файла для конкретного компьютера.
rem Файл заранее создаётся командой djoin /provision на контроллере домена.
set ODJ_FILE=%~dp0SKLAD1.txt

rem Применяем offline domain join к установленной Windows.
rem %~dp0 означает папку, из которой запущен этот .cmd, то есть флешку.
djoin /requestodj /loadfile "%ODJ_FILE%" /windowspath C:\Windows /localos

rem Перезагружаем ПК, чтобы доменное присоединение вступило в силу.
shutdown /r /t 0
```

Запуск:

```cmd
D:\join.cmd
```

Плюс такого варианта: не нужно вшивать букву флешки в скрипт, потому что `%~dp0` берёт путь запуска.

## Универсальный PowerShell-скрипт для ODJ

Если хочется чуть удобнее, можно сделать `join-odj.ps1`, который сам берёт файл рядом со скриптом.

```powershell
# Папка, из которой запущен скрипт.
# На флешке это будет путь к флешке, например D:\.
$ScriptDir = Split-Path -Parent $MyInvocation.MyCommand.Path

# Имя ODJ-файла для этого компьютера.
# Для каждого ПК файл обычно свой: SKLAD1.txt, SKLAD2.txt и т.д.
$OdjFile = Join-Path $ScriptDir "SKLAD1.txt"

# Проверяем, что файл существует.
# Без него djoin не сможет применить offline join.
if (-not (Test-Path $OdjFile)) {
    Write-Error "ODJ file not found: $OdjFile"
    exit 1
}

# Применяем offline domain join к текущей установленной Windows.
djoin /requestodj /loadfile $OdjFile /windowspath C:\Windows /localos

# Перезагружаем ПК сразу после успешного применения.
shutdown /r /t 0
```

Запуск:

```powershell
Set-ExecutionPolicy Bypass -Scope Process -Force
D:\join-odj.ps1
```

## Что хранить на флешке

Минимальный набор:

```text
join.cmd
SKLAD1.txt
SKLAD2.txt
SKLAD3.txt
```

Или для PowerShell-варианта:

```text
join-domain.ps1
join-odj.ps1
SKLAD1.txt
```

## Что проверить после перезагрузки

На клиенте:

```cmd
whoami
hostname
ipconfig /all
nltest /dsgetdc:lab.loc
gpupdate /force
gpresult /r
```

На контроллере домена:

```cmd
dsa.msc
```

Проверить:

- компьютер появился в нужной OU;
- клиент видит DC;
- DNS указывает на доменный DNS;
- GPO применяются.

## Частые ошибки

### Клиент не видит домен

Почти всегда проблема в DNS.

```cmd
ipconfig /all
nslookup lab.loc
nltest /dsgetdc:lab.loc
```

DNS должен указывать на контроллер домена, а не на внешний DNS вроде `8.8.8.8`.

### Неправильная OU

Проверить LDAP-путь:

```text
OU=Workstations,OU=Kaluga,DC=lab,DC=loc
```

Если OU называется иначе или находится в другом месте, join упадёт.

### Неверное имя компьютера

ODJ-файл создаётся под конкретное имя:

```cmd
/machine SKLAD1
```

Если использовать один и тот же файл на нескольких ПК, начнутся конфликты имён.

### Время сильно отличается от DC

Kerberos чувствителен ко времени.

```cmd
w32tm /query /status
```

### Скрипт не запускается

Запустить PowerShell с временным bypass:

```powershell
Set-ExecutionPolicy Bypass -Scope Process -Force
```

Это действует только в текущем окне PowerShell.

## Что лучше использовать

Для лаборатории и редких установок:

```text
join-domain.ps1
```

Для подготовки нескольких рабочих мест:

```text
djoin + join.cmd
```

Главное различие:

```text
Add-Computer - нужен доменный логин и пароль на клиенте
djoin       - пароль на клиенте вводить не нужно
```
