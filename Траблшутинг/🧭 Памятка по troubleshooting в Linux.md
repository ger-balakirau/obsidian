# 1. Логика: от большего к меньшему

Основной принцип — **двигаться сверху вниз по стеку и снаружи внутрь**.

## 🔄 Универсальная цепочка

```
Инфраструктура → Сеть → Хост → ОС → Процесс → Сервис → Зависимости → Данные
```

## 📍 Как применять

### 1. Инфраструктура

- Жива ли машина / VM / instance?
- Есть ли доступ (SSH)?

**Команды:**

- `ping`
- `ssh`
- доступ через cloud / hypervisor

---

### 2. Сеть

- Доступен ли хост?
- Открыт ли порт?

**Команды:**

- `ip a`
- `ip route`
- `ping`
- `traceroute`
- `nc -zv host port`
- `ss -tulnp`

---

### 3. Хост (ресурсы)

- Хватает ли CPU / RAM / Disk?

**Команды:**

- `top`, `htop`
- `free -h`
- `df -h`
- `uptime`

---

### 4. ОС

- Есть ли системные ошибки?

**Команды:**

- `dmesg`
- `journalctl -xe`

---

### 5. Процесс

- Запущен ли сервис?

**Команды:**

- `ps aux | grep <service>`
- `systemctl status <service>`

---

### 6. Сервис

- Слушает ли порт?
- Есть ли ошибки в логах?

**Команды:**

- `ss -tulnp`
- `journalctl -u <service>`
- `tail -f /var/log/...`

---

### 7. Зависимости

- БД, API, кеш, внешние сервисы

**Команды:**

- `curl`
- `nc`
- клиенты БД (`psql`, `mysql`)

---

### 8. Данные / конфигурация

- config файлы
- env переменные

**Команды:**

- `cat`
- `env`
- `diff`

---

# 3. Фильтры и приёмы (обязательно освоить)

## 🔍 grep — база всего

- Поиск:
    
    - `grep error file.log`
- Без регистра:
    
    - `grep -i error`
- Исключение:
    
    - `grep -v debug`
- Контекст:
    
    - `grep -A 5 -B 5 error`

---

## 🔗 Комбинации (pipeline)

- Ошибки сервиса:
    
    ```bash
    journalctl -u nginx | grep -i error
    ```
    
- Топ процессов:
    
    ```bash
    ps aux --sort=-%cpu | head
    ```
    
- Тяжёлые файлы:
    
    ```bash
    du -sh * | sort -hr | head
    ```
    

---

## 🎯 awk — быстрый анализ

- Колонки:
    
    ```bash
    ps aux | awk '{print $1, $2, $3}'
    ```
    
- Фильтр:
    
    ```bash
    df -h | awk '$5 > 80'
    ```
    

---

## 📈 sort / uniq

- Частые ошибки:
    
    ```bash
    grep error log | sort | uniq -c | sort -nr | head
    ```
    

---

## ⏱ tail / less

- Реальное время:
    
    ```bash
    tail -f log
    ```
    
- Поиск внутри:
    
    ```bash
    less log
    /error
    ```
    

---

# 4. Практический алгоритм (как думать)

## 🔹 Шаг 1 — Симптом

Пример: "сервис не отвечает"

## 🔹 Шаг 2 — Проверка по слоям

1. Хост доступен?
2. Порт открыт?
3. Процесс жив?
4. Логи есть?
5. Зависимости работают?

---

## 🔹 Шаг 3 — Сужение

Не делай всё сразу.  
Иди по дереву:

```
если нет сети → не трогаем сервис
если сервис не запущен → не трогаем БД
```

---

# 5. Частые паттерны (запомнить)

## ❗ Connection refused

→ сервис не слушает порт

## ❗ Timeout

→ сеть / firewall / routing

## ❗ 502 / 503

→ upstream (backend умер)

## ❗ Killed

→ OOM killer

## ❗ Permission denied

→ права / SELinux

# 7. Главное правило

**Сначала факты — потом гипотезы.**

Типичная ошибка новичка:

- сразу “перезапущу всё”

Правильный подход:

- понять _почему_ сломалось → потом чинить
